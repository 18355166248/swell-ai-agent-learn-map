# Dev Copilot 演示脚本

## 演示目标

用 5-8 分钟让观众理解三件事：

1. 这不是普通聊天机器人，而是一个会自主调用只读工具的研发分析 Agent
2. 它能同时利用代码和文档两类信息源
3. 当前版本的能力边界和下一步计划都很清楚

## 演示前准备

### 环境检查

在仓库根目录确认：

```bash
cat .env | rg OPENAI_API_KEY
```

如果使用默认免费模型，演示稳定性会差一些。正式分享建议优先使用 `gpt-4o` 或其他稳定模型。

### 启动步骤

```bash
cd projects/04-dev-copilot
npm test
npx tsx src/server.ts
```

启动成功后打开：

- Web UI: `http://localhost:8083`
- 健康检查: `http://localhost:8083/api/health`

## 推荐演示顺序

### Demo 1：先展示“它会自己找信息”

适合第一段的任务：

```text
分析这个项目有哪些工具函数，它们分别解决什么问题
```

预期观察点：

- 会先列目录或直接搜索工具注册表
- 会继续读取 `registry.ts`、`readFile.ts`、`searchDocs.ts` 等文件
- 最终答案会归纳出 5 个只读工具和职责划分

这段的目的是让观众先建立心智：它不是凭空回答，而是在真实看代码。

### Demo 2：展示“代码 + 文档”双信息源联动

推荐任务：

```text
如果我要分析一个真实需求，除了代码，还会用到哪些内部文档能力？
```

预期观察点：

- 会调用 `search_docs`
- 能说明文档索引来自前面的 RAG 项目
- 会把代码工具和知识库工具的角色分开讲

这段的重点不是答案本身，而是让观众看到系统不是单一信息源。

### Demo 3：展示“Agent 和 RAG 的区别”

推荐任务：

```text
解释这个项目里 RAG 和 Agent 分别负责什么
```

预期观察点：

- 会定位 `search_docs`
- 会结合 `runAgent()` 的工具循环做解释
- 输出会自然区分“知道什么”和“先查什么”

这段和分享 PPT 的核心观点一致，适合作为中段理论收束。

## CLI 备用方案

如果现场浏览器网络、SSE 或页面状态不稳定，直接切到 CLI：

```bash
cd projects/04-dev-copilot
npx tsx cli.ts --model openai/gpt-4o "分析这个项目有哪些工具函数"
```

CLI 的优点：

- 更稳定
- 更容易看清工具调用顺序
- 不依赖浏览器状态

## 讲解话术建议

### 开场

> 这个项目不是想做一个泛化 Copilot，而是想回答一个更具体的问题：前端研发在分析需求时，怎么同时利用代码和内部规范，而不是只靠经验。

### 讲工具

> 第一版只做只读工具，这是刻意约束。因为在没有把边界和评估做清楚之前，先让模型“会看、会查、会分析”，比让它“直接改代码”更重要。

### 讲 RAG

> `search_docs` 不是重新造轮子，而是直接复用前面几周的 RAG 能力。Agent 在这里负责编排，不负责重做知识库。

### 讲边界

> 当前版本最重要的工程判断，是路径安全、大文件读取边界、工具结果截断和缓存失效。这些不是附属逻辑，而是 Agent 真正能不能落地的基础。

## 演示中的风险提示

### 风险 1：免费模型不稳定

现象：

- 不调用工具直接回答
- 工具调用顺序不理想

应对：

- 现场优先使用更稳定模型
- 准备 CLI 兜底
- 分享时明确说明模型稳定性不是当前项目完全可控的变量

### 风险 2：知识库索引未准备好

现象：

- `search_docs` 返回“文档索引尚未建立”

应对：

- 演示前确认 `02-doc-rag/.data/vectors.json` 和 `03-req-analyst/.data/vectors.json` 存在

### 风险 3：分享时只讲“成功路径”

应对：

- 主动说明这版仍然是只读 Agent
- 说明尚未建立系统性的需求分析评估集
- 把“下一步做写操作 Agent”留到后续阶段，不在这次演示里夸大
