# Chunk 策略对比实验

## 实验设置

- 知识库：5 个 Markdown 文档（auth-flow, component-library, deploy-cicd, project-react-rules, upload-cdn）
- 评估集：10 条问题（eval-dataset.json），覆盖简单事实查询、流程规则、跨文档综合
- 对比维度：chunk 数量、大小分布、关键词覆盖率

## 策略 A：固定大小切分（基线）

按双换行切段落，合并短段直至接近 500 字符阈值。

## 策略 B：按标题切分

按 Markdown 标题（# / ## / ###）建立层级，每个标题段作为独立语义块。超长段（> 800 字符）按段落二次拆分。

## 对比结果

| 指标         | 固定大小切分 | 按标题切分 | 变化  |
| ------------ | ------------ | ---------- | ----- |
| 总块数       | 29           | 63         | +117% |
| 平均大小     | 412 字符     | 184 字符   | -55%  |
| 中位数大小   | 441 字符     | 144 字符   | -67%  |
| 关键词覆盖率 | 90.9%        | 90.9%      | 持平  |

### 按文档分布

| 文档                   | 固定大小 | 按标题 |
| ---------------------- | -------- | ------ |
| auth-flow.md           | 3        | 8      |
| component-library.md   | 5        | 8      |
| deploy-cicd.md         | 3        | 9      |
| project-react-rules.md | 15       | 29     |
| upload-cdn.md          | 3        | 9      |

## 分析

### 关键词覆盖率持平的原因

两种策略覆盖相同的原始文本，关键词必然存在于某个 chunk 中。关键词覆盖率只能衡量「是否包含」，无法衡量检索精度。

### 按标题切分的优势（理论上）

1. **语义边界更清晰**：每个 chunk 对应一个标题段，检索时能精准定位到相关小节
2. **块更小更聚焦**：平均 184 字符 vs 412 字符，减少噪声
3. **上下文保留**：标题前缀随 chunk 一起存储

### 按标题切分的代价

1. **块数量翻倍**：63 vs 29，向量存储和检索计算量增加
2. **极端短块**：最小 38 字符（如单行标题），可能缺乏足够语义信息
3. **长表格/代码块**：目前未特殊处理，可能被切断

## 推荐

短期保留两种策略，在 rag.ts 中支持切换。长期可结合两种策略做混合切分——先用标题切分作为一级边界，过短块合并，过长块二次拆分。

# Query 改写对比实验

## 实验设置

- 使用固定大小切分的索引（29 chunks，vectors.json）
- Top-3 检索，对比改写前后的 expectedSource 命中率
- 10 条评估用例

## 策略 A：直接嵌入原始问题（基线）

```typescript
embedding = getEmbedding(question);
results = retrieve(embedding, vectors, (topK = 3));
```

## 策略 B：原文 + 改写词混合嵌入

```typescript
expanded = rewriteQuery(question); // 3-5 个关键词
searchQuery = `${question} | ${expanded}`;
embedding = getEmbedding(searchQuery);
results = retrieve(embedding, vectors, (topK = 3));
```

改写 Prompt：要求输出 3-5 个领域关键词，不超过 40 字。

## 对比结果

| 指标          | 原始问题 | 混合改写 | 变化  |
| ------------- | -------- | -------- | ----- |
| 总命中率      | 86.7%    | 90.0%    | +3.3% |
| simple 命中率 | 77.8%    | 77.8%    | 持平  |
| medium 命中率 | 83.3%    | 91.7%    | +8.3% |
| hard 命中率   | 100%     | 100%     | 持平  |

### 逐题详情

| 题目                      | 原始 | 改写 | 变化 |
| ------------------------- | ---- | ---- | ---- |
| Q1 access_token 有效期    | 2/3  | 2/3  | ➡    |
| Q2 前端压缩图片默认参数   | 2/3  | 2/3  | ➡    |
| Q3 InfiniteScroll API     | 3/3  | 3/3  | ➡    |
| Q4 refresh_token 过期处理 | 3/3  | 3/3  | ➡    |
| Q5 灰度发布流程           | 2/3  | 3/3  | ⬆ +1 |
| Q6 ImageViewer SSR 报错   | 2/3  | 2/3  | ➡    |
| Q7 自定义 Hook 规范       | 3/3  | 3/3  | ➡    |
| Q8 上传图片完整流程       | 3/3  | 3/3  | ➡    |
| Q9 TailwindCSS 规范       | 3/3  | 3/3  | ➡    |
| Q10 CDN 缓存策略          | 3/3  | 3/3  | ➡    |

## 分析

### 混合策略有效的关键

1. **原文保留语义核心**：原始问题通过 `|` 分隔符保留，确保主语义不被稀释
2. **改写词补关键词盲区**：补充文档中出现的领域术语，弥合"用户表达 vs 文档措辞"的 gap
3. **控制改写长度**：限制 3-5 词、40 字以内，防止关键词过载反噬嵌入精度

### 第一版失败的原因

初版 Prompt 允许 30-80 字、不限关键词数量，导致输出 20+ 个泛化术语，稀释原始语义。Q2 命中率从 2/3 降到 0/3 就是典型——改写词中包含大量无关术语，将检索结果引向了无关文档。

### medium 难度提升最明显

medium 类问题涉及流程/规则，用户表达往往简略，改写补充的领域关键词能有效帮助检索定位到正确段落。

## 推荐

- **默认启用**：混合策略无退化风险，可作为 rag.ts 默认行为
- **改写词数量硬限制**：Prompt 中明确要求 3-5 个关键词，配合 max_tokens 约束
