# Chunk 切分策略实验

## 实验目的

找出最适合工程文档（Markdown 规范、SDK 说明、接口文档）的切分策略。

## 评估数据集

见 `eval-dataset.json`，包含 10 个问题 + 预期答案 + 预期来源。

## 策略对比

### 策略 A：固定大小切分（500 字）

**实现方式**：每 500 个字符切一刀，有 50 字的重叠（overlap）。

**优点**：实现简单，chunk 大小均匀。

**缺点**：会在句子中间切断，破坏语义完整性。

**适合**：快速原型，对准确率要求不高的场景。

### 策略 B：按 Markdown 标题切分

**实现方式**：遇到 `##` 或 `###` 就切一刀，保留标题作为 chunk 的第一行。

**优点**：保持语义完整，每个 chunk 对应一个完整主题。

**缺点**：chunk 大小差异很大（有的标题下只有 1 行，有的有 50 行）。

**适合**：文档类内容（规范、指南、教程）。

### 策略 C：按语义单元切分（接口/代码块）

**实现方式**：遇到代码块（` ``` `）、接口定义、配置项就作为一个完整 chunk 保留。

**优点**：技术文档中接口名+入参+出参+示例在同一 chunk 里，检索效果最好。

**缺点**：需要自定义解析规则，实现复杂。

**适合**：接口文档、SDK 文档、配置规范。
