# Agent / RAG 评估实验

> 这一组实验不再回答“功能有没有做出来”，而是回答“系统到底有没有变好”。

## 这组实验关注什么

从 Phase 2A 开始，实验重点从“检索策略对比”扩展到“任务完成质量评估”。

这里会逐步积累两类东西：

- 任务集
- 评估结果

## 评估对象

### 1. RAG 类任务

关注：

- 检索到的来源是否正确
- 引用是否准确
- 回答是否和问题真正相关

### 2. Agent 类任务

关注：

- 最终任务是否完成
- 是否调用了正确工具
- 工具顺序是否合理
- 是否触发了不该触发的边界行为

## 第一版评估维度建议

| 维度             | 说明                 |
| ---------------- | -------------------- |
| `retrieval_hit`  | 是否命中预期来源     |
| `citation_ok`    | 引用来源是否正确     |
| `task_completed` | 是否完成了任务本身   |
| `tool_path_ok`   | 工具调用路径是否合理 |
| `constraint_ok`  | 是否遵守系统边界     |
| `notes`          | 人工备注             |

## 失败分类建议

| 类型                | 含义                     |
| ------------------- | ------------------------ |
| `retrieval_miss`    | 没找到正确知识           |
| `citation_wrong`    | 找到了但引用错了         |
| `tool_choice_wrong` | 用错工具或顺序明显不合理 |
| `task_incomplete`   | 回答不完整，任务未完成   |
| `constraint_break`  | 触发了不该触发的边界问题 |

## 文件建议

- `task-set-template.json`：第一版任务集模板
- 后续可新增：
  - `rag-eval-round-01.json`
  - `agent-eval-round-01.json`
  - `report-round-01.md`

## 使用原则

- 先少量、高质量任务，不要一开始就追求数量
- 先人工判分，再考虑自动裁判
- 每次 Prompt / 检索 / Agent 调整后，都应该能回到这里做回归
