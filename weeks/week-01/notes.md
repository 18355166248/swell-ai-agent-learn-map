# Week 1 学习笔记

> 记录学到的东西、踩的坑、值得复用的代码片段。

## 核心概念

### System Prompt vs User Prompt

System Prompt 用来定义“模型长期扮演什么角色、遵守什么规则、输出遵循什么边界”。
它更像系统级约束，优先级高，适合放这些内容：

- 角色设定：你是代码分析助手 / 需求分析助手
- 输出边界：只输出 JSON，不要解释
- 质量要求：优先给引用来源，缺信息时明确说不知道
- 风险约束：不要编造接口、不要猜测未提供的上下文

User Prompt 用来表达“这次具体要做什么”。它更像一次性的任务输入，适合放这些内容：

- 本次要分析的文件内容
- 本次提问的问题
- 本次希望重点关注的角度，例如风险、依赖、组件

当前对 AI 应用最重要的理解：

- System Prompt 决定“这个助手长期怎么工作”
- User Prompt 决定“这次具体做什么”
- 两者配合，才能既稳定又灵活

一个简单模板：

```text
System:
你是一个前端代码分析助手。
你的任务是阅读输入代码，输出结构化 JSON。
不要编造不存在的接口或组件。

User:
请分析下面这个 TSX 文件，告诉我：
1. 这个文件做了什么
2. 依赖了哪些接口或 hook
3. 有哪些组件
4. 有哪些潜在风险
```

### JSON 输出约束

在工程场景里，结构化输出比“回答得像人”更重要，因为后续可能要：

- 在 CLI 里直接打印
- 存进文件
- 作为下一步 Agent 的输入
- 做自动校验或渲染

因此从一开始就应该要求模型返回稳定 JSON，而不是自由文本。

实践时需要注意：

- 先定义字段，再写 Prompt
- 字段名尽量稳定、语义单一，不要一会儿叫 `risk` 一会儿叫 `risks`
- 提前约定空值策略，例如没有命中时返回空数组 `[]`
- 不要让模型同时做“自然语言发挥”和“严格结构化”两件冲突的事

推荐的最小输出结构：

```json
{
  "summary": "一句话总结文件职责",
  "dependencies": [],
  "components": [],
  "risks": []
}
```

Prompt 里要明确写清楚：

```text
请严格返回 JSON，不要输出 Markdown，不要加解释性前缀。
如果某一项没有识别到内容，返回空数组。
```

Week 1 的结论：

- 先把输出结构设计清楚，再写 Prompt
- JSON 结构越稳定，后面做 CLI、Web UI、Agent 链路越轻松
- “结构化输出”本质上是在降低后处理成本

### Function Calling / Tool Calling

Week 1 只需要先理解概念，不急着实现。

当前理解：

- 普通问答：模型只能“说”
- Tool Calling：模型可以根据任务需要“决定调用工具”
- 工具本身不神秘，本质就是程序提前注册好的函数，例如：
  - `read_file(path)`
  - `search_code(query)`
  - `search_docs(query)`

这对后面的意义是：

- RAG 负责让模型“知道什么信息相关”
- Agent + Tool Calling 负责让模型“按步骤做事”

Week 7 再深入实现即可，Week 1 先建立这个心智模型。

### Temperature 的作用

Temperature 控制输出的“发散程度”。

- 低 temperature：更稳定、更保守、更适合结构化任务
- 高 temperature：更发散、更有创造性、更适合文案或头脑风暴

对当前学习项目的结论：

- 代码分析、需求分析、RAG 问答，优先低 temperature
- 生成标题、写文案、想多种方案时，才适合适当调高

在这个仓库的预期场景里：

- `0 ~ 0.3`：适合 JSON 输出、代码解释、引用型问答
- `0.4 ~ 0.7`：适合需要一定表达变化但不能太飘的输出
- `0.8+`：不适合做严肃工程输出，容易漂移

补一个一起理解的概念：

- `max_tokens`：控制最多生成多少内容，避免输出过长或费用失控
- `context window`：模型一次能看到的总上下文大小，包含输入和输出

所以不是“把文档全塞进去就行”，而是要从一开始就意识到上下文预算是稀缺资源。

### 第一个 API 调用要关心什么

从工程角度看，第一次调通 API 不只是“能返回字符串”，而是要同时确认：

1. 请求参数怎么组织
2. 返回结构怎么解析
3. 出错时如何定位
4. 输出是否适合后续程序消费

对后续 CLI 项目，最小闭环应该是：

```text
读取文件
→ 拼 Prompt
→ 调模型
→ 拿到结构化结果
→ 打印到命令行
```

## 踩坑记录

### 1. 太容易先写 Prompt，后想输出结构

如果先写 Prompt，再临时决定返回字段，最后往往会出现：

- 字段名反复变化
- 某些字段时有时无
- CLI 或下游解析逻辑很难写

当前修正方式：

- 先定义 JSON schema
- 再按 schema 倒推 Prompt

### 2. 学习文档容易写成“计划”，不是“结论”

如果笔记里全是“我打算做什么”，过一周后几乎没有复用价值。

当前修正方式：

- 每个主题至少写一条“我现在的结论是什么”
- 尽量写成未来可复用的判断标准或模板

### 3. 容易把 AI 应用理解成单次聊天

真正工程化以后，核心不是“跟模型聊得顺不顺”，而是：

- 输入能不能稳定组织
- 输出能不能稳定消费
- 出错能不能定位和兜底

这也是为什么本周重点应该放在 Prompt、结构化输出和最小 CLI 闭环，而不是先追复杂框架。

### 4. “能编译”不等于“真实跑通”

这次收尾时，已经确认了：

- 根仓库 `npm test` 通过
- `projects/01-ai-code-explain` 的 `npm run build` 通过
- `dist/cli.js` 可以正常启动，并进入真实请求前的校验流程

但在当前 Codex 执行环境里，工作区源码默认不能直接发到外部模型服务，所以还不能把这一步算成“真实模型已跑通”。

这个区别后面要一直保留：

- 本地验证：编译、测试、命令入口、参数解析
- 端到端验证：真实输入、真实模型、真实输出

只有第二类做完，才能把“已可运行”写得更硬。

这次收尾里，真实模型端到端验证已经由本机终端补做完成，所以 Week 1 可以正式从“收尾”切到“完成”。

## 值得复用的 Prompt 片段

### 代码总结 Prompt

```text
你是一个前端代码分析助手。

请阅读用户提供的代码文件内容，并严格返回 JSON。

输出字段要求：
- summary: 一句话总结文件职责
- dependencies: 文件依赖的接口、hook、store、工具函数
- components: 使用到的组件名
- risks: 潜在风险、边界情况、异常处理缺失点

要求：
1. 不要输出 Markdown
2. 不要添加 JSON 之外的解释
3. 如果某项没有内容，返回空数组
4. 不要编造不存在的依赖
```

### 面向真实文件的 User Prompt 模板

```text
请分析下面这个文件，重点关注：
1. 这个文件做了什么
2. 它依赖了哪些接口或外部能力
3. 它渲染了哪些主要组件
4. 是否存在潜在风险

文件路径：
{{filePath}}

文件内容：
{{fileContent}}
```

### 当输出不稳定时的修正提示

```text
请重新输出，并满足以下要求：
1. 只返回合法 JSON
2. 字段必须包含 summary、dependencies、components、risks
3. dependencies、components、risks 必须是数组
4. 不要输出任何额外解释
```
