# Week 7 学习笔记：Agent + Tool Calling

## 这周真正做成了什么

Week 7 的核心不是继续增强 RAG，而是让模型从“知道答案”升级为“会自己找答案”。本周完成的 `projects/04-dev-copilot/` v1 已经具备一个最小可用的只读 Agent：

- 用户给出一个研发分析任务
- 模型根据工具描述决定下一步动作
- 程序执行工具，把结果再喂回模型
- 模型继续判断是否需要更多工具，直到给出最终答案

这说明仓库的能力链路已经从 Prompt → API → RAG，推进到了 Agent + Tool Calling。

## Tool Calling 的实现骨架

这一版实现遵循最朴素也最稳定的模式：

1. 定义工具 schema
2. 把 tools 一起发给模型
3. 读取 `tool_calls`
4. 在本地执行工具
5. 把工具结果作为 `role: "tool"` 消息补回上下文
6. 继续下一轮，直到得到最终答案

对应到项目结构：

- `src/agent/index.ts`：ReAct 主循环
- `src/agent/tools/registry.ts`：工具定义与执行入口
- `src/agent/tools/*.ts`：各个只读工具
- `src/server.ts` / `cli.ts`：两种调用入口

## 这周落地的 5 个工具

| 工具          | 作用                       | 关键限制                                     |
| ------------- | -------------------------- | -------------------------------------------- |
| `read_file`   | 读取文件内容，支持行号范围 | 限制项目根目录内访问，大文件必须显式给行范围 |
| `list_files`  | 递归列目录，支持 glob 过滤 | 跳过无关目录，避免结果爆炸                   |
| `search_code` | 关键词搜索代码             | 只扫常见代码文件类型                         |
| `grep`        | 正则搜索并返回上下文       | 更适合精确定位局部模式                       |
| `search_docs` | 检索知识库文档             | 复用 `doc-rag` 能力，并带缓存失效策略        |

设计上坚持“第一版只读，不直接改代码”，这是本周最重要的安全边界。Agent 可以分析，但不允许直接写文件或执行高风险操作。

## ReAct 循环里最重要的几个点

### 1. 迭代轮次和工具步数不是一回事

一轮模型响应里可能包含多个 `tool_calls`。因此项目里把：

- `iterations` 定义为 Agent 主循环轮数
- `steps` 定义为 Thought / Action / Observation 级别的记录

这样后面做 UI 和调试时，语义更稳定，不会把“一次回答里调了两个工具”误算成两轮思考。

### 2. 工具结果不能无限长

工具返回结果最终还是要喂给模型，所以必须控长。实现里对工具结果做了截断，避免某次文件读取或代码搜索把上下文直接塞爆。

### 3. 大文件读取必须有硬边界

`read_file` 是这周最容易做坏的工具。开始时只解决了“不要直接整文件读入”，后来又继续收紧成：

- 小文件可以直接读到末尾
- 大文件如果不指定 `endLine`，直接拒绝
- 大文件只有明确行范围时才允许读取

这里的关键不是“能不能读”，而是要保证 Agent 不会因为一次不受控调用把上下文和内存打爆。

### 4. 文档检索要兼顾复用和实时性

`search_docs` 没有自己重写一套检索逻辑，而是直接复用 `02-doc-rag` 的公共接口。后面又补了基于 `mtime` 的缓存失效，避免两种问题：

- 每次调用都全量读向量索引，导致热路径变慢
- 索引更新后一直读旧缓存，结果不生效

## 这一周的工程取舍

### 为什么先做只读工具

因为 Week 7 的目标是理解 Agent 调用链，而不是构建一个全自动改代码机器人。先把“观察环境、读取信息、组合分析”的链路做稳定，才能在 Week 8 再考虑更强能力。

### 为什么不用复杂 Agent 框架

这一版直接基于 OpenAI SDK + function calling 手写循环，原因很明确：

- 逻辑透明，容易看懂
- 更适合学习 Tool Calling 的原理
- 出问题时调试成本低
- 目前 5 个工具的复杂度不需要引入 LangChain 一类抽象层
