## 本周想回答的问题

- Agent 如何记住上一轮聊了什么？
- 如何在不过度消耗上下文窗口的前提下注入历史？
- 记忆的边界在哪里（什么该记、什么不该记）？

## 核心理解

### Memory 的本质不是"存起来"，是"在正确的时候放回上下文"

Agent 本身是无状态的——LLM 每次调用只看当前的 messages 数组。所以"记忆"的实现本质上是：

1. **存**：对话结束后，把关键信息持久化到外部存储
2. **取**：下一轮对话开始时，把历史信息注入到 messages 数组的合适位置
3. **用**：System Prompt 告诉 LLM 这段历史的含义和如何使用

不需要修改 ReAct 循环本身——这是最干净的设计。

### 为什么存回答而不是存完整 messages

完整 messages 里包含了 tool_calls 和 tool results。Tool results 可能非常大（8000 字符截断），存几轮就几十 KB。更关键的是，注入完整历史会让 messages 数组迅速膨胀，挤占上下文窗口。

而 Agent 的最终回答已经是精炼结论——"分析结论：共 5 个工具..."——它天然就是一份摘要。用回答作为记忆，既省空间又保持了信息密度。

### 历史注入的位置很重要

```text
messages = [
  system prompt,      ← 角色定义（永远第一）
  memory context,     ← 对话历史（system 角色，语义上是"已知信息"）
  user task,          ← 当前问题
  preloaded context,  ← （可选）预取的工具清单
]
```

把历史放在 system prompt 之后、user task 之前，LLM 会把它视为"已有的背景知识"而非"需要执行的新指令"。如果放在 user task 之后，模型容易困惑"这是之前的事还是现在的事"。

### Memory 的粒度选择

当前设计：**每轮对话存一个 turn**。每个 turn 包含：

- 用户问题（完整）
- 助手回答（完整，最多 600 字用于注入上下文）
- 时间戳
- 工具步骤摘要（可选，用于调试）

不存中间的工具调用细节——它们只在该轮有意义，跨轮引用价值低。

## 对 Week 11 的判断

评估体系（Week 9）和记忆（Week 10）都有了。下一步应该是：

**Human-in-the-loop（人机协同）** — 让 Agent 在不确定时主动向用户确认，而不是硬着头皮编造。

这会涉及：

- Agent 输出 "我需要确认" 的能力
- 前端展示确认卡片（不是纯文本）
- 用户确认后 Agent 继续执行
- 评估体系中增加 "是否在不确定时主动确认" 的维度
