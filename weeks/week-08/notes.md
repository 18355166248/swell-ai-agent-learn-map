# Week 8 笔记：8 周演进复盘，而不是单项目收尾

## 最终阶段判断

Week 8 最终完成的不是某一个额外功能，而是把前 7 周的学习路径收束成一条完整演进线：

Prompt → LLM API → Embedding → RAG → 检索优化 → 真实场景 → Agent → 总结与分享。

这一周最终落下的核心产物是：

- `docs/learning-summary.md`：8 周学习总结
- `weeks/week-08/ppt/index.html`：技术分享网页 PPT

## 为什么分享材料要现在做

如果 Week 8 只围绕 `dev-copilot` 讲，会有两个偏差：

- 会把“8 周学习路线”缩窄成“最后一个项目”
- 会弱化 RAG 概念、检索优化和阶段演进本身的学习价值

所以 Week 8 的材料应该服务于整条学习路径。它需要回答的是：

- 每一周学了什么
- 每一周的项目产物是什么
- 为什么会从上一周自然推进到下一周
- 哪些概念是真正通过做项目学会的

## 分享结构选择

这次分享材料仍然使用 `Guizang Ppt Skill` 的瑞士国际主义风格，但内容重点要调整：

- 主题不是单工具发布，而是 8 周技术演进
- 内容重点是阶段变化、概念关系、方法判断和踩坑
- 瑞士风更适合做“阶段复盘 + 方法总结”型分享

最终版本保留 IKB 克莱因蓝主题，并把“时间线”“阶段认知变化”“RAG 与 Agent 的衔接关系”放到了分享稿中心。

## 这周最终完成了什么

1. 把 8 周总结从“项目清单”收成了“演进叙事”
2. 调整了 PPT，让每一页都明确服务于“8 周学习路径”
3. 只保留必要的项目说明，不再让 `dev-copilot` 占据叙事中心
4. 补上了“概念是怎么一步步学会的”和“周与周如何衔接”的表达

## 这周的判断标准

Week 8 结束时，至少要满足：

- 文档状态和代码状态一致
- 有 8 周总结
- 有一份围绕“8 周进度和演进步骤”的分享材料
- 能清楚讲出 RAG 和 Agent 分别是在第几周、为什么出现、解决了什么问题

现在这个标准已经满足。Week 8 可以被定义为完成，因为它已经把“项目、概念、阶段、判断”四条线收到了同一套材料里。
