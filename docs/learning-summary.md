# 8 周学习总结：从 Prompt 到 RAG，再到 Agent 的演进路径

## 一句话总结

这 8 周的重点不是做出某一个单独工具，而是顺着一条连续工程路径，把 AI 应用里最关键的几层能力真正做一遍、串起来、讲清楚：

Prompt → LLM API → Embedding → RAG → 检索优化 → 真实需求分析 → Agent + Tool Calling → 分享与复盘收束。

最终的几个项目只是这条学习路径的阶段性载体。真正要总结的，是“每一周为什么往下走、每一步学会了什么、能力是怎么演进出来的”。

## 8 周演进地图

| 周次   | 学习重点                               | 真正学会的概念   | 对应产物             |
| ------ | -------------------------------------- | ---------------- | -------------------- |
| Week 1 | Prompt / 基础调用                      | 模型输入输出约束 | `ai-code-explain v1` |
| Week 2 | Streaming / 上下文 / 错误处理          | LLM 工程可用性   | `ai-code-explain v2` |
| Week 3 | Embedding / Chunk / Cosine             | 向量检索基础     | `doc-rag` 核心模块   |
| Week 4 | RAG Pipeline / 引用来源 / Web UI       | RAG 闭环         | `doc-rag` Web Demo   |
| Week 5 | Query Rewrite / BM25 / Hybrid Retrieve | 检索优化方法     | 检索实验与回填       |
| Week 6 | 真实规范文档 / 结构化分析              | RAG 落地真实场景 | `req-analyst v1`     |
| Week 7 | Tool Calling / ReAct / 工具边界        | Agent 编排能力   | `dev-copilot v1`     |
| Week 8 | 复盘 / 总结 / 分享                     | 把学习路径讲清楚 | 总结文档 + 分享材料  |

## 10 分钟分享提纲

如果要把这 8 周压缩成一次短分享，最适合按下面 5 段来讲：

1. 起点是什么：不是为了做某一个工具，而是想把 AI 应用工程里最关键的几层能力真正做一遍。
2. 前 2 周学会了什么：先把 Prompt、API、Streaming、上下文和错误处理做成稳定可用的最小闭环。
3. Week 3 到 Week 6 为什么重要：这几周不是泛泛而谈 RAG，而是把 Embedding、Chunk、检索、优化、真实文档场景逐步压实。
4. Week 7 为什么是拐点：Agent 不是替代 RAG，而是建立在前面知识库和代码观察能力之上的下一层编排。
5. Week 8 最终要交付什么：把“每一周解决了什么、为什么会进入下一周、最后是怎么汇合的”讲清楚。

如果要再压缩成一句话，就是：

不是“8 周做了 4 个项目”，而是“8 周把一个 AI 应用从最小调用一路做到了 RAG、真实场景和 Agent 编排”。

## 为什么要按这条路线学

一开始就上复杂 Agent 框架或多智能体编排，学习成本高，而且很容易陷入“会调概念，不会落项目”。这套学习路线刻意从最小闭环开始：

1. 先会稳定调用模型
2. 再处理输入输出和上下文边界
3. 再做检索与知识库
4. 最后让模型具备调用工具和分步分析能力

这样的好处是每一周都有一个明确可运行的产物，而不是只有笔记和想法。

## 周与周之间是怎么接起来的

- Week 1 到 Week 2：先证明模型能调用，再解决“怎么稳定地用”。
- Week 2 到 Week 3：文本一长、文档一多，单次 prompt 不够了，必须进入 embedding 和检索。
- Week 3 到 Week 4：只有向量和相似度还不够，需要做成完整 RAG 闭环和可演示产品。
- Week 4 到 Week 5：RAG 能用之后，瓶颈马上变成检索质量，所以必须做实验和优化。
- Week 5 到 Week 6：优化过的 RAG 要接到真实研发文档里，才知道它是否真的有业务价值。
- Week 6 到 Week 7：当问题从“问一个知识点”变成“做一个分析任务”，就需要 Agent 来决定先查什么、再看什么。
- Week 7 到 Week 8：Agent 做完不代表路线结束，还要回过头把整条演进路径讲清楚，才能形成真正可复用的方法论。

## 8 周分别做成了什么

### Week 1：把最小 LLM 调用闭环跑通

- 理解 system prompt / user prompt / JSON 输出约束
- 做出 `projects/01-ai-code-explain` v1
- 建好整个学习仓库结构和测试基线

这一周的关键不是功能复杂度，而是建立“先做能跑通的最小项目”的习惯。

### Week 2：补齐工程可用性

- 支持 streaming
- 加入长文档处理和上下文控制
- 支持 `--file` / `--dir`
- 加入错误处理和重试

到这一步，LLM 调用才从“demo”变成“可以稳定用的命令行工具”。

### Week 3：理解向量化与检索

- 使用 embedding 向量化文本
- 手写 cosine similarity
- 实现按标题切分 Markdown
- 建立最基础的向量检索链路

这周最重要的收获是：RAG 的门槛不在“调用 embedding API”，而在“怎么切、怎么存、怎么查、怎么验证”。

### Week 4：完成第一版 RAG Demo

- Express 服务
- 文件上传
- Web UI
- 引用来源展示
- 端到端可用的问答体验

到这时，知识库问答已经从 CLI 工具变成了一个完整的可演示服务。

### Week 5：开始做效果优化，而不是只求“能用”

- 建立评估数据集
- 对比不同 chunk 策略
- 加入 query 改写
- 加入 BM25
- 做混合检索和 RRF 融合

这周最大的认知变化是：RAG 质量问题不能靠直觉调，必须通过实验和指标来收敛。

### Week 6：接入真实研发文档场景

- 收集组件库、埋点、接口、支付、发布等真实规范
- 建立真实知识库索引
- 完成结构化需求分析输出
- 做出 `projects/03-req-analyst`

这一步把“泛 RAG”推进成了“和前端研发工作流相关的 RAG”。

### Week 7：让模型从“知道”变成“去做”

- 实现 Agent ReAct 循环
- 接入 5 个只读工具
- 增加 CLI 和 Web UI 的多步分析链路
- 完成 `projects/04-dev-copilot` v1

这周是整个路线里最关键的一跳。RAG 解决的是“知道什么”，Agent 解决的是“接下来该查什么、先看什么、怎么把信息拼起来”。

### Week 8：整合、收尾、分享

Week 8 的目标不是继续堆新功能，而是做三件事：

- 把已有项目讲清楚
- 把学习收获写清楚
- 把最终产物整理成可以展示的版本

当前这份总结文档和 `weeks/week-08/ppt/index.html` 就属于这一阶段的交付物。

## 为什么 Week 8 不应该只讲最终项目

如果 Week 8 只讲 `dev-copilot`，会丢掉前面 6 周最重要的学习脉络：

- RAG 的概念不是在 Week 7 才突然出现的
- 检索优化不是最后项目里的一个附属特性，而是 Week 5 的核心学习内容
- Agent 也不是终点，它只是建立在前面 Prompt、检索、真实场景经验之上的下一层能力

所以 Week 8 的总结应该优先讲：

1. 8 周分别解决了什么问题
2. 每一周和下一周之间的递进关系
3. 哪些概念是通过项目真正学会的
4. 最终项目如何承接前面的能力

## 最终项目在 8 周里的位置

最终项目 `projects/04-dev-copilot/` 的确重要，但它更适合被定义为“前 7 周能力的汇合点”，而不是 Week 8 唯一主角。

它对应的真实问题是：

- 新人或跨模块同学不知道先看哪份文档
- 需求分析时容易漏掉接口、埋点、规范限制和测试点
- 代码和文档是两套信息源，人工切换成本高
- 纯聊天式问答经常停留在“说得像”，但不能落到文件和规范

对应能力拆分为：

- 文档侧：`search_docs`
- 代码侧：`search_code` / `grep` / `read_file` / `list_files`
- 编排侧：Agent ReAct 循环
- 交互侧：CLI + SSE Web UI

## 这 8 周里最有价值的技术判断

### 1. 先做透明系统，再做高级抽象

在 Agent 阶段没有直接引入复杂框架，而是手写循环。这让工具调用、消息追加、失败回退和结果截断都清晰可见，后面调试边界问题时成本很低。

### 2. 检索质量来自实验，不来自感觉

从 chunk 策略、query 改写到 BM25 融合，真正有效的是“实验比较 + 回填最优方案”，而不是单靠经验说哪种方案更强。

### 3. 安全边界必须在工具层落实

到了 Agent 阶段，最危险的不是“模型答错”，而是“模型调用了一个边界不清的工具”。所以 Week 7 对路径安全、大文件读取、二进制检测、只读限制这些边界做得比功能本身更重要。

### 4. 复用比重写重要

`dev-copilot` 没有重新做一套 RAG，而是直接复用 `doc-rag` 的能力。这是项目从学习 demo 走向工程产物的关键一步。

## 踩过的坑

### 1. 文档状态和代码状态很容易失真

如果每次改完代码不同步更新 README / PROGRESS / 周文档，很快就会出现“代码已经变了，文档还停在两周前”的情况。后面几周明显把这件事做得更严格了。

### 2. 大文件读取边界比想象中更难

`read_file` 最开始只是在“能读”和“不能读”之间切换，后来才收敛成真正可控的策略：大文件必须显式行范围，小文件才允许整段读取。

### 3. 缓存策略不能只考虑性能

`search_docs` 一开始用模块级缓存，后来又发现索引更新后读不到新内容。最终变成“有缓存 + 按 mtime 失效”，这是性能和正确性的折中。

### 4. 测试通过不等于语义正确

多次 review 都说明一个问题：单纯“测试全绿”不能说明没有回归。只有把关键语义，比如 `silent` 是否真的静默、Agent 轮次如何计数、文件读取边界是否真正封死，都写进测试，测试才有价值。

## 现在再看，最重要的收获是什么

不是“我会 RAG 了”，也不是“我做了一个 Agent”。更重要的是这几件事：

- 知道如何把一个 AI 概念压缩成可运行的最小工程闭环
- 知道什么时候该做实验，什么时候该做约束
- 知道 AI 工程最怕的不是复杂，而是边界模糊
- 知道最终产物必须服务真实工作流，而不是只服务技术演示

## 如果继续往下做，下一阶段最值得补的是什么

### 1. 结果质量评估体系

现在已经有项目原型，但还缺一套稳定的需求分析评估数据集和打分标准。

### 2. 写操作的安全方案

Week 7 刻意只做只读工具。如果后面要进入“辅助改代码”阶段，需要先定义写操作的审批机制、回滚策略和最小权限边界。

### 3. 更真实的团队场景验证

比如：

- 不同类型需求分析是否真的省时间
- 是否能降低漏埋点、漏接口、漏规范的概率
- 新人 onboarding 是否能更快进入状态

### 4. 更完整的展示材料

当前已经有可用的网页 PPT 最终版，但如果要做对外或对内正式分享，还可以继续补：

- 一次完整 demo 录屏
- 对比纯人工分析的案例
- 技术架构图精修版
- 失败案例与限制说明

## 结尾

这 8 周最大的意义，不是把每个名词都学一遍，而是把“AI 应用工程”真正走了一遍。

从最早的 prompt 到最后的 Agent，本质上一直在回答同一个问题：

怎么把模型能力变成一个前端研发场景里真正有用、可演示、可复盘、可继续演进的系统。
